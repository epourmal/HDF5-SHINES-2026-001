\subsection{Footprints, Fingerprints, Controls -- HDF5 Privacy}

HDF5 privacy is best treated as an exposure-management problem, not only as a
confidentiality or access-control problem. In a broader privacy-engineering
sense, ordinary data processing can create privacy harm through inference,
aggregation, secondary use, contextual change, or surveillance effects even when
systems are secure and operating as designed. That distinction matters for HDF5
because the format was built to make scientific data portable, self-describing,
reusable, and long-lived. Those same properties can preserve or reveal
information that a data producer did not intend to disclose.

In scientific settings, sensitive information may appear outside the primary
dataset values. HDF5 files often cross institutional boundaries, enter public
repositories, move into cloud workflows, and survive long enough to be reused
with analysis techniques that did not exist when the files were created.
Machine-learning and cross-repository analysis make this more acute: information
that appears harmless in one file can become revealing when correlated with
quality-control products, timestamps, geospatial metadata, workflow records, or
external data sources.

\subsubsection{Privacy-Sensitive Assets}

The privacy surface of an HDF5 ecosystem includes several asset classes.

\begin{description}
    \item[Data values.]
    Datasets may contain biomedical observations, wearable-sensor measurements,
    geospatial products, industrial telemetry, facility observations, images,
    simulations, or other measurements that directly disclose sensitive
    information or enable sensitive inference.

    \item[User metadata.]
    HDF5 groups, datasets, committed datatypes, dataspaces, attributes, object
    names, link names, compound datatype member names, enumeration labels, and
    object comments can expose context. Timestamps, coordinates, instrument
    identifiers, acquisition settings, calibration details, experimental
    identifiers, and provenance fields are scientifically useful, but may also
    identify people, facilities, locations, schedules, or research objectives.

    \item[Structural metadata.]
    The hierarchy of groups and datasets, dataset dimensions, chunking,
    filtering pipelines, external storage, links, object references, region
    references, and virtual datasets can reveal categories, processing stages,
    cohorts, workflow assumptions, and relationships among observations even
    when raw values are not inspected.

    \item[Internal format metadata.]
    HDF5 library and file-format structures can also leak clues. Examples
    include global heaps that store variable-length data, object reference
    counts, textual datatype descriptions produced by applications, creation
    order tracking, chunk indexing structures, and other implementation details
    that reveal the shape or semantics of stored content.

    \item[Derived and reconstructed information.]
    Privacy-sensitive facts may be inferred from combinations of datasets,
    linked files, external links, region references, virtual dataset mappings,
    masks, quality-control arrays, and provenance records. In these cases the
    protected asset is not a single stored object, but a relationship that can
    be reconstructed.

    \item[Workflow artifacts.]
    Tool output, \texttt{h5dump} reports, diagnostic logs, temporary files,
    intermediate products, metadata snapshots, conversion artifacts, and
    plugin-generated metadata may persist after the workflow that created them.
    These artifacts can expose information that users do not associate with the
    released HDF5 file itself.
\end{description}

\subsubsection{Exposure Mechanisms}

Five exposure mechanisms recur in HDF5 workflows.

\begin{description}
    \item[Metadata exposure.]
    Because HDF5 is self-describing, metadata remains visible and useful across
    tools. That visibility can reveal acquisition time, location, instrument
    identity, laboratory conditions, patient class, operational status, or
    internal annotations even when primary data values are transformed,
    anonymized, or protected elsewhere.

    \item[Structural inference.]
    File structure often mirrors scientific intent. Paths, group organization,
    dataset dimensions, chunking choices, compression filters, external storage
    mappings, and virtual dataset relationships can disclose experimental
    design, processing stages, classifications, or the absence or presence of
    expected observations.

    \item[Raw-representation exposure.]
    Low-level storage patterns and convenience metadata may reveal information
    about raw content without requiring full application-level interpretation.
    Fixed-length strings, textual datatype descriptions, descriptive
    attributes, and implementation-specific layout choices can expose semantics
    even when direct value access is restricted.

    \item[Unintended inclusion or alteration.]
    Converters and processing pipelines frequently enrich data to improve
    interoperability. Well-intentioned conversion from non-self-describing
    formats into HDF5 can turn implicit structure into explicit datasets or
    metadata. Derived products and provenance added for reproducibility can
    become new privacy surfaces.

    \item[Cross-data linkage.]
    HDF5 supports linkage across files and systems through external links,
    external storage, virtual datasets, VFDs, and VOL connectors. These features
    are valuable for science, but can also make otherwise benign measurements
    revealing when combined with geospatial data, timestamps, masks,
    quality-control arrays, or external repositories.
\end{description}

\subsubsection{Lifecycle Controls}

Privacy in HDF5 is not a static property of a file. It changes as data are
collected, converted, annotated, transformed, shared, archived, replicated, and
reused. A privacy-aware HDF5 workflow should therefore treat exposure review as
part of the data lifecycle.

\begin{itemize}
    \item Classify datasets and metadata before release, including attributes,
    names, datatype fields, dimensions, VDS mappings, references, and external
    links.
    \item Minimize metadata that is not needed for the intended reuse, and
    replace sensitive names, labels, locations, identifiers, and timestamps with
    controlled or generalized values where scientific meaning permits.
    \item Review conversion and enrichment tools for metadata they add
    automatically, especially when translating from less self-describing formats
    into HDF5.
    \item Separate sensitive provenance, operational notes, logs, and temporary
    workflow products from public data products unless they are intentionally
    part of the release.
    \item Treat structure as inspectable information. Redaction plans should
    consider object paths, hierarchy, dimensions, chunk layout, filters, links,
    references, and related side files, not only dataset values.
    \item Reassess older HDF5 archives when data are moved to new repositories,
    combined with other collections, opened to AI/ML workflows, or released to a
    broader audience than originally expected.
\end{itemize}

\subsubsection{Control Levels}

The practical control posture falls along a continuum.

\begin{description}
    \item[Open by design.]
    Some HDF5 data products are intended to be fully inspectable. For these
    releases, the control is explicit documentation: state what metadata,
    structure, provenance, links, and workflow artifacts are intentionally
    exposed, and verify that no unintended sensitive context remains.

    \item[Exposure-reduced sharing.]
    The middle ground is usually the most important case. Preserve enough HDF5
    metadata for reproducibility and interoperability, but remove, generalize,
    tokenize, or separate fields that identify people, facilities, precise
    locations, operational schedules, unpublished scientific objectives, or
    sensitive workflow details.

    \item[Strong protection.]
    When either values or metadata cannot be safely exposed, use access control,
    encryption, repository policy, or container-level protections around the
    file and its side artifacts. Encryption of raw values alone is insufficient
    if names, attributes, structure, logs, links, or external storage mappings
    still reveal the sensitive context.
\end{description}

\subsection{HDF5 Tools}

Not strictly part of the core library, but the tools are an important part of the ecosystem and have their own set of issues.

\subsection{Audit Findings and Mitigation Priorities}

\begin{itemize}
    \item HDF5 privacy review should cover user-controlled metadata and internal
    structural metadata in addition to raw datasets.
    \item Tooling should help users inventory attributes, names, references,
    external links, VDS mappings, filters, datatype fields, and workflow
    artifacts before publication.
    \item Documentation and training should teach HDF5 users to recognize
    exposure surfaces, because many privacy failures arise during legitimate
    sharing and reuse rather than from malicious access.
    \item Privacy guidance for HDF5 should be aligned with open-science goals:
    the objective is not to make data opaque by default, but to make exposure
    intentional, reviewable, and proportionate to the release context.
\end{itemize}
